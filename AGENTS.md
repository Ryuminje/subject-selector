# 고교학점제 선택과목 수요조사 및 분반 배정 자동화 시스템 (Subject Selector)

## 📌 프로젝트 소개
이 프로젝트는 **고교학점제 선택과목 수요조사 데이터를 기반으로 시수 산출, 시간표 작성, 분반 배정까지의 과정을 자동화**하는 웹 기반 시스템입니다.
교사가 엑셀 데이터를 업로드하면 즉각적으로 교육과정 편제와 시간표를 작성하고, 학번순 분반 등 번거로운 수작업을 시스템이 자동으로 처리합니다.
이 문서는 추후 이 프로젝트를 유지보수하거나 새 기능을 추가할 **AI 에이전트(Agent)**를 위한 가이드(Rules/Instructions) 역할을 합니다.

## 🛠 주요 기능 및 워크플로우

### [수요조사 기반 시수 산출 탭]
**1단계: 교육과정 편성표 업로드 및 교과(군) 수동 설정**
- 엑셀 파일을 업로드하여 예비1학년/2학년/3학년별 전체 과목 편제를 파싱합니다.
- **동적 열 탐색:** 엑셀 양식이 다르더라도 '구분', '교과(군)', '과목', '운영학점'을 동적으로 스캔합니다.
- **특수 기호 분리:** 과목명에 ↔ 기호가 있는 경우(예: 물리학Ⅰ↔화학Ⅰ) 두 과목 분리.
- **학기별 운영학점 산출:** '운영학점' 열을 무시하고 실제 학기 아래에 적힌 숫자를 가져와 학점으로 간주합니다.
- **교과(군) 인라인 수정:** 자동 추출된 교과군이 정확하지 않을 경우 사용자가 표 안에서 즉시 텍스트를 클릭하여 수동으로 덮어쓸 수 있으며 이는 전 단계에 연동됩니다.

**2단계: 수요조사 파일 업로드**
- 학생들이 제출한 수강 희망 과목 엑셀 데이터를 파싱합니다.

**3단계 & 4단계: 분석 및 위계성 검사**
- 학생별 선택 학점 수(시수)가 조건에 맞는지 확인합니다.
- **원천 데이터 기반 교과군 판별:** 하드코딩된 규칙 없이, 오직 1단계에서 업로드한 교육과정 엑셀 원본의 교과군 기준만을 참고해 기초/사회/과학/기타 학점을 정확히 집계합니다.
- **타겟 학년(Off-by-one) 자동 보정:** 학생들이 '현재' 학년이 아닌 '내년'에 들을 과목을 신청하는 점을 감안해, 학번 기준(2학년=1학년생, 3학년=2학년생)으로 정확하게 코호트를 매핑합니다.

**5단계: 과목 개설 여부 및 개설 반 수 판별**
- 수요조사 통계를 기반으로 학급 기준 인원과 비교하여 개설 반 수를 자동 추천합니다.
- **수동 오버라이드 지원:** 자동 계산된 반 수를 클릭하여 원하는 정확한 반 수로 수동 덮어쓰기 할 수 있습니다. 이 데이터는 백업 시 완벽히 보존됩니다.
- **개설 상태 자동 판별:** 수동/자동으로 계산된 반 수가 '딱 떨어지는 숫자'인 경우 **[확정]**, 논의 등 문자인 경우 **[미정]**, 10명 미만인 경우 **[폐강]**으로 직관적인 배지를 띄워 줍니다.
- **데이터 초기화 연동:** 3단계 파일을 삭제하면 4단계뿐 아니라 5단계의 통계 데이터도 즉각적으로 초기화됩니다.

**6단계: 교과(군)별 시수 정리표 (선택과목 자동 연동)**
- 1단계의 지정과목과 5단계에서 **[확정]** 판정을 받은 선택과목을 교과군별로 그룹화하여 다년도 시수 통계를 하나의 표로 병합해 보여줍니다.
- **개설 반 수 우선순위 제어:** 선택과목의 시수 계산 시 무조건 상단의 '전체 반 수'를 따르지 않고, **5단계에서 수동으로 입력/확정된 개설 반 수를 최우선으로 사용하여** 정확한 과목별 시수를 자동 계산합니다. (물론 6단계 표 안에서 또 다시 수동으로 수정할 수 있으며, 이 경우 6단계 데이터가 가장 최우선시 됩니다)
- **수석교사 감축 시수 실시간 연동:** 수석교사가 있는 교과군을 선택하여 감축 시수를 입력하면 해당 교과의 1학기 및 2학기 '교과별 총 시수'에서 즉시 차감됩니다. 입력 UI에서 데이터 변경 시 [적용] 버튼 없이 실시간으로 계산되며, 차감 내역은 직관적으로 '(원래시수-감축시수)' 형태로 부가 표기됩니다.
- **교사 수 기반 평균 시수 도출:** 각 교과별 첫 줄의 '교사 수'를 수동으로 입력받아, 운영학점과 개설 반 수를 통해 도출된 총 시수(감축 시수가 반영된 찐 시수 기준)를 교사 수로 나눈 평균 시수(1년/학기당)를 자동 제공합니다.
- **총계 산출:** 6단계 테이블 최하단에 '총계' 행을 두어 1학기, 2학기, 1년에 대한 과목별 시수 총합 및 교과별 총 시수의 총합을 자동으로 렌더링하며 이 과정에서도 감축 시수가 모두 반영된 결과를 나타냅니다.

### [선택과목 변경 탭]
**1단계: 학생 선택 데이터 업로드 (타임별 데이터)**
- 열 헤더가 과목명이고 각 셀이 A, B, C 등 선택 그룹(타임)을 나타내는 **시간표 편성용 수요조사 엑셀(sample3.xlsx 형태)**을 파싱합니다.

**2단계: 타임별 시간표 입력 및 과거 이수 과목 업로드**
- 2학년/3학년별로 A, B, C 등 타임별 시간표(행)와 각 반(열)을 관리하는 동적 Grid(격자)를 제공합니다.
- 격자의 각 칸은 위(과목명), 아래(교사명)로 나뉘어 있으며, 엑셀 복사/붙여넣기를 완벽하게 지원합니다.
- **학년별 독립적인 데이터 업로드:** 과거 이수 과목 데이터(2학년 수강과목, 3학년 1학기 과목) 업로드 시, 2학년과 3학년 데이터가 각각 독립된 상태 객체(`Record<GradeKey, ...>`)로 분리 저장되어 서로 덮어쓰지 않도록 설계되어 있습니다.

**3단계: 타임별 선택과목 명단 산출 (변경 전 자동 분반 배정)**
- 1단계의 학생 데이터와 2단계의 시간표를 결합하여 과목별/반별 학생 명단을 산출합니다.
- **강력한 자동 분반 기능:** 만약 시간표 Grid에 동일한 과목이 여러 반으로 나뉘어 기재될 경우(예: 심리학1, 심리학2, 심리학 3반), 정규식을 이용해 원본 과목명(심리학)을 자동 추출합니다.
- 추출된 과목을 선택한 전체 학생을 학번순(오름차순)으로 정렬한 뒤, 반 개수에 맞춰 정확히 등분하여 배정합니다.

**4단계: 선택과목 변경 신청 내역 및 자동 검증**
- 학생별로 **과목 변경 내역(변경 전 과목 -> 변경 후 과목)**을 입력받습니다. 여러 과목을 변경하는 학생을 위해 `+` 버튼을 통해 묶음 입력을 지원합니다.
- **상/하단 그리드 분리 운영:** 실제 학생의 '변경 신청 입력(신청자)'과 교사가 개입하는 '인원 균등 분배를 위한 임의 변경' 그리드가 분리되어 운영됩니다. 두 데이터는 별도의 상태(`electiveChanges`, `electiveChangesArbitrary`)로 관리되나 내부 알고리즘(`adjustmentLog`) 연산 시 병합되어 순차적으로 교환/검증을 수행합니다.
- **결과 시각적 분리:** 우측 결과 그리드에서는 상단(신청자 결과)과 하단(임의 변경 결과)이 시각적으로 명확하게 나뉘어 렌더링되며, 실시간으로 연동되어 표시됩니다.
- **자동 교체 로직(2단계 우회 교환 지원):** 변경 후 과목이 변경 전 과목과 동일한 타임에 있다면 1단계 교체를 즉시 승인합니다. 만약 다른 타임에 있다면, 해당 과목이 개설된 모든 타임을 스캔하여 다른 빈 타임으로 밀어내는 형태의 **2단계 우회 교체(Swap)**가 가능한지 검증합니다.
- **인원 균등화 최적화 알고리즘:** 타임별 과목 인원수가 최대한 고르게 분포되도록 Hill Climbing 기반의 시뮬레이션을 수행하여 최적의 교체 경로를 도출하는 기능을 제공합니다.
- 불가피하게 교환이 불가능한 경우 빨간색 `(불가)` 태그와 함께 불가능한 사유(예: "B타임에 개설되지 않음")를 툴팁으로 안내하여 직관적인 피드백을 제공합니다.
- 로마 숫자(Ⅰ, Ⅱ, Ⅲ)와 영문 알파벳(I, II, III), 띄어쓰기를 자동으로 정규화하여 비교하는 Fuzzy Matching 기능이 내장되어 있습니다.

**5단계: 변경 후 타임별 선택과목 명단 산출**
- 4단계에서 검증 완료(Success)된 변경 내역만을 바탕으로 학생 명단을 재배정하여 최종 결과를 보여줍니다.
- 변경된 학생의 이름은 초록색으로 강조되며 상세한 툴팁을 제공합니다.

**6단계: 다년도 수강 내역 위계 및 분석**
- 1단계에서 별도 업로드한 과거 수강 내역과 2학기 최종 과목 배정 내역을 결합하여 학생의 전체 선택 과목 트리를 분석합니다.
- **자동 위계성 검증:** 수요조사 탭의 2단계에서 설정한 '위계성 규칙(Hierarchy Rules)'을 공유하여 과거와 현재 수강 과목들 사이의 선수/후수 과목 이수 순서가 올바른지 완벽히 교차 검증합니다.
- **과목군 제약 엄격 검사:** 1단계 기초자료에서 가져온 교과(군) 데이터 기준과 정확히 매칭하여 기초, 사회, 과학 과목의 이수 개수 및 학점을 필터링합니다.

### 엑셀 명단 다운로드 및 백업 파일(JSON) 기능
- **타임별 다중 시트 생성:** 명단은 A, B, C 등 모든 타임을 순회하여 하나의 엑셀 파일 내 독립적인 탭(시트)으로 완벽하게 분할하여 저장합니다.
- **동적 셀 병합 및 스타일링:** 변경전/후 사이에 화살표(→) 열을 삽입하고, 헤더 굵기 조정, 가운데 정렬, 얇은 테두리가 일괄 적용됩니다.
- **백업 파일(JSON) 호환성 보장:** 과거에 저장한 백업 파일을 불러올 때 키(학번) 값에 띄어쓰기 공백이 포함되어 있더라도, 동적으로 `.trim()` 정규화를 수행한 뒤 상태에 반영하여 이전 데이터와 최신 시스템이 100% 호환되도록 합니다. 모든 수동 오버라이드(5단계, 6단계 반 수 조작 데이터)는 빠짐없이 JSON에 시리얼라이즈되어 유지됩니다.

## 💻 기술 스택 및 구현 세부사항
- **Framework:** Next.js (TypeScript, React)
- **Styling:** Tailwind CSS (다크 모드 베이스, Glassmorphism, 모던 프리미엄 UI 지향)

## ⚠️ AI 에이전트를 위한 향후 개발 가이드라인 (Rules)

1. **상태 관리 및 학년 분리:** 
   - 예비1/2학년과 3학년 데이터(`changeActiveGrade`)는 UI 상의 뷰(View)뿐만 아니라 내부 상태(과거 이수 과목 데이터, 백업 내역 등) 역시 반드시 `Record<GradeKey, ...>`와 같은 형태로 각각 완벽하게 분리된 상태로 유지보수해야 합니다.
2. **분반 정규식 로직 절대 유지:** 
   - `page.tsx` 내의 명단 산출 로직 중 `cleanBase`와 `cleanChosen`을 통해 모든 공백과 기호를 무시하고 매칭하는 로직을 함부로 축소하지 마세요. 이는 다양한 사용자 입력값(타이포그래피 오타, 띄어쓰기 다름)을 보정하기 위한 필수 로직입니다.
3. **2단계 교환 검증 로직 완벽 탐색:** 
   - 4단계의 `adjustmentLog`는 반드시 하나의 과목만 찾는 것이 아니라 `findSlotsWithSubject`를 통해 과목이 개설된 모든 타임을 전부 탐색하여 가능한 Swap 경우의 수를 모두 시도해야 합니다.
4. **연쇄적 상태 업데이트(Working Copy):** 
   - 한 학생이 여러 과목을 동시에 변경할 수 있으므로, 4단계 교환 로직에서 `studentSchedules`와 같은 '임시 시간표(Working 연산용 임시 복사본)'를 만들어 변경이 발생할 때마다 실시간으로 시간표 상태를 갱신해야 충돌 에러가 발생하지 않습니다.
5. **원본 데이터 보존 원칙:** 
   - 교육과정 편제 파싱 시 로마자(Ⅰ, Ⅱ)를 아라비아 숫자로 강제 변환하지 않고 엑셀 원본 그대로 표기합니다.
6. **UI/UX 기준 및 레이아웃:** 
   - 기능 추가 시 Tailwind CSS를 이용해 **직관적이고 미려한 UI(hover 애니메이션, 트랜지션, 색상 조화)**를 필수로 유지하세요. 단순한 형태의 테이블이나 버튼은 지양합니다.
   - 정보량이 많은 그리드(Grid)나 데이터 비교 화면을 구성할 때는 화면 가로 비율을 최대한(예: 2:1 비율 등)으로 사용하여 가로 스크롤과 줄바꿈을 최소화하고 한눈에 직관적으로 파악할 수 있는 넓은 레이아웃을 유지하세요.
7. **백업 및 불러오기 안정성 유지:**
   - 시스템의 상태 구조가 변경되더라도, 기존에 저장된 `JSON` 백업 파일과의 하위 호환성(Backwards Compatibility)을 항상 고려하여 파싱 로직(`handleLoadBackup`)을 보수적으로 짜야 합니다. 모든 수동 입력 상태는 반드시 Backup State에 포함시켜 영속성을 부여해야 합니다.
   - 백업 파일 저장 시, 최신 웹 표준인 `File System Access API`(`window.showSaveFilePicker`)를 사용하여 사용자가 저장 위치와 파일명을 직접 지정할 수 있도록 지원하며, 미지원 브라우저를 대비한 Fallback 로직을 반드시 포함합니다.
8. **엑셀 내보내기 서식 세밀화:**
   - 학생 명단 다운로드 시 단순 데이터 출력이 아닌, '과목명', '강의실', '교사', '순번' 등의 구체적인 헤더와 행을 유지하고 모든 셀에 테두리를 적용하며 명단 최하단에 '총 인원' 통계를 포함하는 등 사용자 친화적인 엑셀 서식을 엄격히 유지합니다.
9. **업로드 UI 상태 보존:**
   - 파일이 업로드된 이후에도 사용자에게 해당 업로드 칸의 목적(설명, 제목)을 숨기지 않고 계속 노출해야 합니다. 또한 실제 업로드된 파일명(확장자 포함)을 명시해주고 원상태로 복구할 수 있는 '삭제' 기능을 필수적으로 제공해야 합니다. (이때 삭제 액션은 하위 파생 데이터를 연쇄적으로 전부 삭제해야 함을 명심하세요)
10. **예외 처리 및 안전장치 강화:**
   - Hill Climbing과 같은 최적화 알고리즘 루프 내에서 특정 학생의 데이터(`optimizedLogs[studentId]`)가 누락될 수 있는 경우를 대비해, 항상 빈 배열로 초기화하거나 `if (optimizedLogs[studentId])` 와 같은 방어 코드를 필수로 작성하여 런타임 TypeError를 방지해야 합니다.
11. **입력 폼 정렬 UX (Jumping 방지):**
   - 사용자 데이터를 입력받는 그리드(Grid)에서 자동 정렬 기능을 구현할 때, `onChange` 이벤트마다 실시간으로 정렬하면 입력 도중 포커스를 잃거나 행이 위아래로 튀는 현상이 발생합니다. 반드시 `onBlur`(포커스 잃음) 이벤트를 활용해 입력을 마친 시점에만 상태가 정렬되도록 사용자 경험(UX)을 세심하게 고려해야 합니다.
12. **복합 상태(Compound State) 렌더링 최적화:**
   - 여러 상태(예: `electiveChanges`, `electiveChangesArbitrary`)가 융합되어 하나의 뷰나 연산 로직(`useMemo`)에 파이프라인되는 경우, 종속성 배열(`Dependency Array`)에 모든 상태를 명시적으로 등록해야만 반응성(Reactivity)이 끊기지 않습니다. 체크박스(Optimization) 토글 상태 등도 빠짐없이 종속성에 포함시켜야 버그를 예방할 수 있습니다.

13. **반응형 테이블 너비 고정 (Dummy Columns):**
   - 동적으로 열(Column) 개수가 변하는 표(예: 과목 필터링)를 렌더링할 때, 	able-fixed 레이아웃과 빈 가짜 열(Dummy Column)을 사용하여 1개의 열만 표시될 때도 기존 최대 너비를 그대로 유지시켜 과도한 늘어짐을 방지하는 세련된 UI를 유지해야 합니다.
